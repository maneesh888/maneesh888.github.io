Local AI Infrastructure

Open Keyboard + LLM Gateway

A local LLM gateway for routing AI requests through controlled client keys, model access rules, and OpenAI-compatible endpoints.

This project supports Open Keyboard, where the keyboard owns the user experience and the gateway owns request routing, client access, model selection, and operational controls.

The gateway validates client keys, applies per-key limits, returns scoped model lists, and forwards OpenAI-compatible chat requests to configured local or companion model backends.

Current status: active local infrastructure. Recent work added a stronger admin console, structured OpenKeyboard operation handling, key-scoped model discovery, Docker preflight checks, and targeted regression coverage.

Technical Highlights

Auth

Client Key Authentication

Validates client keys before model requests reach the proxy layer, with admin workflows for creating, editing, disabling, and testing access.

Limits

Request Controls

Applies per-key request limits, burst settings, model defaults, and allowed-model rules so local tools can share the gateway with clearer boundaries.

Models

Model Routing

Routes OpenAI-compatible chat requests to Ollama-compatible backends and an optional companion model backend without changing the client integration pattern.

Admin

Admin Console And Playground

Includes a responsive admin console for key management plus a key tester/playground for validating live chat and structured writing-operation requests.

Latest Gateway Progress

Admin

Operational Admin Console

The admin surface now supports API key management, model selection, rate-limit settings, responsive navigation, and a playground for testing a selected key.

  • Create, edit, disable, and delete client keys through the admin UI/API.
  • Choose available models from the gateway and keep a manual fallback when needed.
  • Use the key tester to verify live chat behavior for a selected client key.
Contract

Structured Writing Operations

The proxy understands OpenKeyboard operation requests and shapes responses into structured results that the keyboard can render consistently.

  • Supports grammar correction, summarization, rewriting, translation, and continuation operation names.
  • Normalizes correction items, summaries, replacement text, ranges, confidence, category, and warnings where returned by the model.
  • Rejects unsupported or invalid structured requests before forwarding them upstream.
Models

Scoped Model Discovery

Model listing now respects each key's model configuration and allowlist, so clients see the model set they are allowed to use.

  • Returns OpenAI-style model-list responses for client discovery.
  • Supports local Ollama-compatible models and an optional companion model backend.
  • Prevents disallowed model requests before the gateway forwards a chat call.
Ops

Reliability And Deployment Hygiene

Recent work improved gateway development and deployment checks without putting runtime configuration into public website or application code.

  • Docker base-image preflight catches host, registry, and timeout issues before a build is treated as an application failure.
  • The proxy keeps client authentication separate from upstream model calls.
  • Build and targeted test scripts are part of the local verification workflow.

Boundary

Project Boundary

  • Presented as local/self-hosted infrastructure, not a public SaaS product.
  • Production exposure would still require careful hosting, HTTPS, external access controls, and operational monitoring.
  • Runtime configuration stays outside the public website and tracked application examples.
  • Live model behavior remains environment-dependent; deterministic tests cover gateway contracts and UI wiring.
